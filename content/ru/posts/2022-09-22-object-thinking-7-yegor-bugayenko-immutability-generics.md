## Краткое изложение
В седьмом выпуске серии «Object Thinking» Егор Бугаенко рассматривает архитектурные решения на примере своей Java-библиотеки **xsline**, предназначенной для создания конвейеров (pipelines) преобразования XML-документов. Основная тема обсуждения — как совместить строгую **неизменяемость (immutability)** объектов с необходимостью гибкой настройки через **обобщения (generics)**.

Егор вводит метафоры **Shift** (одно преобразование) и **Train** (поезд или конвейер преобразований). Он выступает категорически против мутабельных методов, таких как `add()`, заменяя их на функциональный метод `with()`. Каждый вызов `with()` возвращает новый экземпляр «поезда», что делает объект потокобезопасным и избавляет от побочных эффектов.

Ключевая техническая задача выпуска — использование дженериков для обеспечения типизации при работе с разными входными данными (например, когда мы хотим передать путь к файлу как `String` вместо готового объекта `Shift`). Бугаенко демонстрирует паттерн с использованием промежуточного интерфейса `Temporary` и метода `back()`. Это позволяет API «эволюционировать» в процессе настройки (например, из `Train<String>` обратно в `Train<Shift>`), не нарушая правила, согласно которому у объекта не должно быть публичных методов, не описанных в интерфейсе. Итоговый код становится **декларативным**: он описывает, «что» представляет собой объект, а не «как» его нужно пошагово изменять.

## Ключевые мысли
- **«Объекты — это не корзины с данными, а живые существа. Чтобы они были надежными, они должны быть неизменяемыми».**
- **«Публичные методы, которых нет в интерфейсе — это архитектурная ошибка, ведущая к протеканию абстракции».**
- **«Дженерики позволяют объекту менять свой тип в процессе 'флюентной' настройки, сохраняя при этом строгую проверку компилятора».**
- **«Декларативный код лучше процедурного, потому что он определяет результат, а не последовательность команд».**

## Релевантность: 5/5
Материал крайне важен для разработчиков и архитекторов, стремящихся к чистому коду. Он показывает, как использовать сложные возможности системы типов Java для соблюдения строгих принципов ООП в реальных проектах.
