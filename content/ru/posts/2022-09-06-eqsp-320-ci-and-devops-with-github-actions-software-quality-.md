## Краткое изложение
В этой лекции из своего курса по качеству программного обеспечения Егор Бугаенко рассматривает практические и философские аспекты непрерывной интеграции (CI) и DevOps на примере GitHub Actions. Он переопределяет CI не просто как набор тестов, а как «защитный механизм», предназначенный для предотвращения попадания некачественного кода в основной репозиторий. Основой его подхода является принцип «Read-Only Master»: разработчики никогда не должны пушить код напрямую в ветку master. Вместо этого все изменения должны проходить через строгий автоматизированный конвейер, включающий модульные тесты (Maven), проверки стиля (Xcop), метрики качества (Kulis) и сканирование безопасности (Snyk).

Бугаенко акцентирует внимание на использовании GitHub Actions благодаря конфигурации на базе YAML и бесплатной инфраструктуре для open-source проектов. Он демонстрирует продвинутые возможности, такие как матричные сборки для одновременного тестирования на разных версиях JDK. Значительная часть лекции посвящена «Rulor» — чат-боту, который управляет процессом слияния (merge). Используя команды вроде `@rulor merge`, бот создает изолированную среду, выполняет слияние локально, прогоняет весь набор проверок и только в случае успеха пушит изменения в master. Это переносит ответственность за качество с конкретного программиста на «систему». Если баг доходит до master-ветки, это считается провалом конвейера CI/CD, а не ошибкой человека.

## Ключевые мысли
- «Качество появляется тогда, когда вы намеренно и явно удаляете плохие части».
- «Master священен: никогда не пушьте в него напрямую; пусть роботы защищают его».
- «Вините систему, а не человека: если баг попал в master, значит, ваш CI-конвейер сломан».
- «Каждая строчка кода должна быть привязана к тикету; если нет задачи — не должно быть и кода».
